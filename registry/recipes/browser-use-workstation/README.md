# Browser-Use Workstation

Browser-based automation workstation for DGX Spark using the official `browser-use/web-ui` project.

## What it provides

- Gradio WebUI on port `7788`
- Optional noVNC browser viewer on port `6080`
- Optional direct VNC endpoint on port `5901`
- Optional Chrome DevTools remote debugging port on `9222`
- Browser-use agent task runner with support for multiple LLM providers
- Persistent task outputs, downloads, traces, recordings, and saved WebUI settings under `./data/tmp`

## Default access

- Web UI: `http://localhost:7788`
- noVNC: `http://localhost:6080/vnc.html`
- VNC: `localhost:5901`
- CDP / remote debugging: `localhost:9222`

## Configuration notes

- Add at least one provider credential in `.env` before first use, such as `OPENAI_API_KEY`, `GOOGLE_API_KEY`, `ANTHROPIC_API_KEY`, or `DEEPSEEK_API_KEY`.
- The upstream UI defaults to `DEFAULT_LLM=openai`, but the provider and model can also be changed inside the WebUI.
- `USE_OWN_BROWSER=true` enables browser reuse mode. When using that mode, set `BROWSER_PATH` and optionally `BROWSER_USER_DATA`.
- `KEEP_BROWSER_OPEN=true` keeps the browser session alive between tasks.
- `BROWSER_CDP` can point to an external Chromium-compatible DevTools endpoint if you want the agent to attach to an existing browser.
- `OLLAMA_BASE_URL` is included for local model routing patterns, but you may still need to set the provider and endpoint inside the UI depending on your chosen backend.

## Persistent data

This recipe stores data under:

- `registry/recipes/browser-use-workstation/data/tmp`
- `registry/recipes/browser-use-workstation/data/browser-profile`

The `tmp` mount preserves downloads, traces, recordings, saved settings, and deep research outputs generated by the WebUI.

## Validation scope

Validated in this repository by:

- creating recipe metadata, environment templates, compose definition, and documentation
- aligning the compose build with the official `browser-use/web-ui` upstream Docker image source
- exposing the upstream runtime ports `7788`, `6080`, `5901`, and `9222`
- preserving persistent host mounts for task artifacts and optional browser profile reuse

Not validated here:

- full upstream image build and runtime smoke test on this Windows workspace
- live agent task execution against external LLM providers
- interactive noVNC or VNC session verification

## Risk notes

- The upstream image is large and includes Playwright Chromium, VNC, noVNC, Node.js, and supervisor services, so initial builds can take significant time and disk.
- Browser automation behavior depends on the selected LLM provider, credentials, and target websites.
- Exposing VNC and CDP ports increases local attack surface; keep this recipe on trusted networks.
